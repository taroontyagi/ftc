Design Tokens

Figma Variables →
The Token Foundation

Tokens are the contract between design and code. Get them right once, and everything downstream maintains itself.

🎨
Set up Figma Variables ↗
Create collections for Color, Spacing, Typography, and Radius. Use semantic naming: color/brand/primary, spacing/md, radius/card. Not blue-500 or 16.
🌗
Modes = Themes
Add a Dark mode to your color collection. Every component using token references automatically responds to theme switching — in Figma and in code.
📤
Export → Claude
Use a Figma plugin to export Variables as JSON. Ask Claude: "Convert this to CSS custom properties, a JS token file, and a Tailwind theme extension."
🔗
Reference in prompts
Always tell Claude where your tokens live: "Tokens are in /tokens/index.ts. Use token references, not hardcoded hex values." This is the single biggest quality multiplier.
💡 The payoff: A designer changes color/brand/primary in Figma → export updated JSON → Claude updates every component that references it. One source of truth. No manual grep-and-replace in code.
Figma Prep

Structure Claude
Can Actually Read

🚨 A file full of Frame 1234 and Rectangle 69 is like asking Claude to read your mind. It will try. The result will be confidently wrong.
🏷️
Layer Names = Component API
Name layers semantically: Button/Primary/Large, Card/Product/Hover. Claude reads the slash hierarchy as component structure. This is your component's public API — treat it that way.
📐
Auto Layout ↗ = Flexbox
Every container should use Auto Layout. Horizontal → flexbox row. Vertical → flexbox column. Gap, padding, alignment — all map directly. Manual positioning creates position: absolute spaghetti.
🧩
Variants = Props
A Button with variant=primary/ghost/destructive and size=sm/md/lg becomes a typed TypeScript interface automatically. One Figma component = one React component. No ambiguity.
🎨
Bind Variables Everywhere
Every fill, stroke, and spacing value should reference a Variable — not a raw hex or pixel value. Token-bound designs produce token-bound code. Raw values produce magic numbers.
Code Connect

Code Connect —
Close the Loop

🔗 Code Connect ↗ links your Figma components to the actual code components in your repo. Developers see import Button from '@/components/Button' in Dev Mode — not raw CSS. Claude uses the same mapping to reference your real library.
🔍
Specific Node URLs
Right-click a component in Figma → Copy link. The URL contains a node-id. This gives Claude surgical focus on one component. Never paste a whole-page URL — context becomes noise.
🖼️
Screenshot + Link Combo
Paste both a screenshot AND the Figma link in the same prompt. Screenshot carries design intent and visual feel. Link carries technical data. The combination produces noticeably better output.
📋
Annotations in Figma
Use Figma's native annotation tool to describe interaction behaviour — hover states, scroll behaviour, animation intent. Claude picks these up via MCP and translates them to CSS transitions and JS.
🗂️
Component Documentation
Add descriptions to components via Dev Mode. Claude reads component descriptions when generating code. "This card is used in product listings and should support lazy image loading" → correct implementation.
The Loop

Figma → Claude → Component
Stay in the design chair

1
🎨 Design the component with intent
Name layers semantically. Use Auto Layout. Bind Variables. Create variants for all states (default, hover, focus, disabled, loading). The time you invest here pays back 10x in code quality.
2
📎 Share the node link + screenshot
Right-click the component → Copy link. Screenshot the component. Paste both into Claude with context: "This is a product card for our e-commerce platform. Stack: Next.js + Tailwind. Tokens in /tokens/index.ts."

Component Systems

Building a Real
Design System

🏆 The goal: one Figma component → one code component → one Storybook story → perfect parity. When that loop is tight, design changes ship in hours, not sprints.
🧩
One Component, All States
In Figma: one component set with all variants (size, variant, state). In code: one React component with typed props. Ask Claude to generate both together: "Generate the component + a Storybook story for every variant."
📖
Living Documentation
Storybook stories generated from Figma variants serve as living docs. Designers can reference the story URL instead of hunting in Figma. Developers know exactly what to expect. QA has test cases.
🔄
Design Updates Without Regen
When a component design changes: share the updated Figma link with a diff prompt. Claude updates only what changed — preserving all logic, event handlers, and prop interfaces. Not a full rewrite.
🔌
Design-Code Parity Check ↗
figma-console-mcp's figma_check_design_parity tool compares Figma component specs against code implementation and returns a scored diff. Run it before every PR that touches a design-system component.
Prompting · Anatomy

The Anatomy of
a Good Prompt

Every strong Figma-to-code prompt has five ingredients. Miss one and Claude has to guess. Guessing produces generic output.

❌
Missing ingredients
"Build this component for me."
— no link, no stack, no tokens,
   no pattern, no intent

Claude produces a generic, unstyled component that matches no design system and satisfies no brief.
✅
All 5 present
① figma.com/design/…?node-id=4-23
② Next.js 15, Tailwind, TypeScript
③ Tokens: /tokens/index.ts
④ Pattern: /components/Card.tsx
⑤ All variants · ARIA · Storybook
① The Figma link
Node-specific URL (?node-id=X-Y). Never paste the whole page URL — the context becomes noise.
② Your stack
Framework, CSS approach, language. Without this Claude picks defaults that may not match your codebase at all.
③ Token file path
The single biggest quality multiplier. Claude references your actual tokens instead of hardcoding hex values.
④ Pattern reference
An existing component to follow. Claude matches your file structure, naming conventions, and code style automatically.

Figma → Web Gap

What Figma Can Do
That the Web Can't

⚠️ These aren't Claude's limitations — they're gaps between Figma's rendering model and the web platform. No tool bridges them automatically. Know them before the client demo.
🌈
Advanced Gradients
Figma supports angular, diamond, and complex multi-stop gradients with arbitrary angles. CSS only has linear, radial, and conic. Diamond gradients don't exist in CSS. Claude will approximate — results vary.
🖼️
Image Fills on Shapes
Figma can fill any vector shape with an image — a star, arrow, custom path. On the web, clip-path + background-image can approximate simple shapes but breaks down fast. Complex paths require SVG masking which Claude will attempt but isn't pixel-perfect.
✨
Layer Blend Modes
Figma's Multiply, Screen, Overlay etc. exist in CSS (mix-blend-mode) but the compositing context differs. Results are visually different when background colours change — especially on dark surfaces or over images.
🔲
Inner Shadows & Stroke Alignment
Figma supports inner shadows and strokes positioned inside, outside, or center. CSS box-shadow only approximates inner shadows. Stroke alignment in CSS is always center — inside/outside must be faked with outline or scale tricks.
🌫️
Background Blur
Figma's background blur (frosted glass) maps to CSS backdrop-filter: blur() which works — but only has partial browser support and breaks in stacked elements with overflow: hidden. Test on real devices.
🎭
Vector Effects & Paths
Figma vector effects (noise, distortion, warp) have no CSS equivalent. Complex boolean operations between paths can be output as SVG — but interactive, dynamic, or animated paths need hand-coded SVG or a library like Lottie, GSAP, or Framer Motion.
🔤
Text on a Path
Figma can place text along any curved path. CSS cannot do this. SVG textPath is the only web equivalent — Claude can generate it for static text, but it won't animate or respond to dynamic content the same way.
📦
Linked Assets & Components
Icons, images, and illustrations in Figma are embedded in the file. Claude can reference them by name but cannot export or embed them automatically. You need to export assets separately and provide file paths. SVG icons work best — raster images need an export step.
💡 Rule of thumb: If it's in Figma's Effects panel, uses a non-standard gradient type, or requires a specific blend mode — verify in browser before committing to the design. Catch the gap at design time, not sprint review.

Let Claude Write It
Run npx skills add anthropics/skills --skill skill-creator then type /skill-creator. Claude interviews you, drafts SKILL.md, runs 20 eval queries, and iterates until it triggers correctly. Minutes, not hours.
🔁
Test It Properly
Write 20 test prompts — half should trigger, half shouldn't. Ask Claude to evaluate both sets. If precision is below 80%, tighten the description. Don't skip this. Skills that don't trigger are skills that don't exist.

Commit to .claude/skills/ for your team. Push to GitHub + list on skills.sh ↗. Others install with npx skills add you/repo. Version it. Bump the version field for auto-updates to work.

Ways of Working

Who Does What
Now That Claude Exists?

🤝 There's no single right answer — it depends on your team's trust, size, and who's closest to the decision. But the question is worth having explicitly, not drifting into by accident.
🎨
Model A — Designer vibes, dev integrates
Designer uses Claude to build UI components directly. Dev handles API wiring, state management, auth, and business logic. Components land in the repo already built.

Works when: Designer is comfortable with Claude Code output and basic component structure. Team has clear handoff contract for where UI ends and logic begins.
Breaks when: Designer produces components that don't follow the codebase conventions, or when business logic bleeds into UI components.
⌨️
Model B — Designer designs, dev vibes
Designer stays in Figma. Dev uses Claude Code to implement from the Figma spec — faster than before, with MCP reading the design directly. Designer reviews the output.

Works when: The design system is well-structured (named layers, tokens, variants). Dev and designer have a shared review ritual.
Breaks when: Figma files are poorly structured, so Claude approximates and the dev doesn't catch the drift from spec.
🔄
Model C — Fluid, with clear ownership
Both use Claude. The question isn't who writes the code — it's who owns the component. Whoever's name is on the PR is responsible for it meeting spec.

Works when: Team culture already has strong code ownership and review habits. High trust, fast feedback loops.
Breaks when: "Claude wrote it" becomes the answer to every review comment. Ownership dissolves.

✦ The question that matters most

  • Who reviews the output? Claude is not the last line of defence. Someone with both design and code context needs to own the final check.
  • Where does the design system live? If it's only in Figma, Model A is risky. If it's in code tokens too, Model A becomes viable.
  • What's the team's bottleneck? If dev is the bottleneck, Model A or B accelerates. If design is the bottleneck, Model B with faster review cycles is the play.
  • Don't drift into a model. Have the conversation explicitly. Decide. Revisit in 6 weeks.
